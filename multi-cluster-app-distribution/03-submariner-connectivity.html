Multi-Cluster Application Distribution

This step deploys Submariner via ACM to establish encrypted L3 connectivity between Cluster A (AWS) and Cluster B (Azure). Submariner creates an IPsec tunnel between the clusters, enabling pods and services to communicate across cloud providers without exposing anything to the public internet. This provides the network foundation that Step 3 (Service Mesh) builds on for cross-cluster traffic management.

Prerequisites

  • ACM Hub cluster with workload placement from Step 1 fully operational

  • Both managed clusters (cluster-a and cluster-b) joined to the Hub and members of the production-clusters ManagedClusterSet

  • oc CLI authenticated as cluster-admin on the Hub cluster

  • Contexts renamed as described in the Cluster Setup section

  • Cloud provider credentials available for both managed clusters (see below)

Cloud provider credentials

Submariner needs cloud credentials to configure gateway nodes and open the required firewall rules (NSGs on Azure, security groups on AWS). If ACM provisioned your clusters, these credentials already exist. If the clusters were imported manually, create them before proceeding.

Verify credentials exist:

oc get secret cluster-a-cloud-creds -n cluster-a --context hub
oc get secret cluster-b-cloud-creds -n cluster-b --context hub

If either secret is missing, create it:

Cluster A (AWS):

oc create secret generic cluster-a-cloud-creds -n cluster-a --context hub \
  --from-literal=aws_access_key_id="<AWS_ACCESS_KEY_ID>" \
  --from-literal=aws_secret_access_key="<AWS_SECRET_ACCESS_KEY>"

Cluster B (Azure):

oc create secret generic cluster-b-cloud-creds -n cluster-b --context hub \
  --from-literal=osServicePrincipal.json='{
    "clientId": "<AZURE_CLIENT_ID>",
    "clientSecret": "<AZURE_CLIENT_SECRET>",
    "tenantId": "<AZURE_TENANT_ID>",
    "subscriptionId": "<AZURE_SUBSCRIPTION_ID>"
  }'
These credentials need permission to modify security group rules (AWS) and NSG rules (Azure). The minimum required permissions are documented in the Submariner cloud preparation guide.

Steps to Apply

1. Create the Submariner broker namespace

What: Create the namespace where the Submariner Broker will run on the Hub.

Why: The Broker is the rendezvous point where managed clusters exchange endpoint information. ACM expects the broker namespace to follow the <managedclusterset-name>-broker naming convention.

oc apply -f 02-submariner-connectivity/broker-namespace.yaml --context hub
Verify: Broker namespace created 
oc get namespace production-clusters-broker --context hub
NAME                          STATUS   AGE
production-clusters-broker    Active   10s

2. Deploy the Submariner Broker

What: Create the Broker CR that coordinates cross-cluster discovery and tunnel establishment.

Why: The Broker stores the endpoint information that each managed cluster’s gateway needs to find and connect to its peers. Without the Broker, the clusters have no way to discover each other.

oc apply -f 02-submariner-connectivity/broker.yaml --context hub
Verify: Broker deployed 
oc get broker submariner-broker -n production-clusters-broker --context hub
NAME                AGE
submariner-broker   10s

3. Create the SubmarinerConfig for each managed cluster

What: Create SubmarinerConfig CRs that tell ACM how to prepare each managed cluster for Submariner - including which credentials to use and how many gateway nodes to designate.

Why: Each cloud provider needs different preparation. On AWS, Submariner opens security group rules for IPsec traffic (UDP 4500, UDP 500). On Azure, it opens NSG rules for the same ports. The SubmarinerConfig also designates worker nodes as gateway nodes, which handle the encrypted tunnel endpoints.

oc apply -f 02-submariner-connectivity/submariner-config-cluster-a.yaml --context hub
oc apply -f 02-submariner-connectivity/submariner-config-cluster-b.yaml --context hub
Verify: SubmarinerConfig created for both clusters 
oc get submarinerconfig submariner -n cluster-a --context hub
oc get submarinerconfig submariner -n cluster-b --context hub
NAME          AGE
submariner    10s

4. Deploy the Submariner add-on on each managed cluster

What: Create ManagedClusterAddOn resources to trigger the Submariner installation on both managed clusters.

Why: ACM manages Submariner as an add-on. Creating this resource tells ACM to install the Submariner operator, gateway engine, and route agent on the target cluster. The gateway engine establishes the IPsec tunnel to the peer cluster, and the route agent configures the host networking so pod and service CIDRs are routable across the tunnel.

oc apply -f 02-submariner-connectivity/addon-cluster-a.yaml --context hub
oc apply -f 02-submariner-connectivity/addon-cluster-b.yaml --context hub
Verify: Submariner add-on installed and connected

Wait 2-3 minutes for the add-on to install and the gateways to establish the tunnel.

Check add-on status:

oc get managedclusteraddon submariner -n cluster-a --context hub
oc get managedclusteraddon submariner -n cluster-b --context hub
NAME          AVAILABLE   DEGRADED   STATUS
submariner    True        False

Check the gateway connection on each cluster:

oc get gateways.submariner.io -n submariner-operator --context cluster-a
NAME           HA STATUS
<gateway-node> active
oc get gateways.submariner.io -n submariner-operator --context cluster-b
NAME           HA STATUS
<gateway-node> active

Verify the connection is established:

oc get gateways.submariner.io -n submariner-operator --context cluster-a \
  -o jsonpath='{range .items[*].status.connections[*]}Status: {.status} | RTT: {.latencyRTT.average}{"\n"}{end}'

The output should show Status: connected with a valid RTT, confirming the IPsec tunnel is established between AWS and Azure.

5. Verify cross-cluster connectivity

What: Run a connectivity test from a pod on one cluster to a service on the other cluster, proving the Submariner tunnel works.

Why: Before proceeding to Step 3 (Service Mesh), we need to confirm that the L3 tunnel is operational and that pods on Cluster A (AWS) can reach pods on Cluster B (Azure) and vice versa. This is the foundation that all cross-cluster traffic management depends on.

Deploy a test pod on Cluster A and try to reach a service on Cluster B:

oc run submariner-test --image=registry.access.redhat.com/ubi9/ubi-minimal:latest \
  -n demo-app --context cluster-a --restart=Never --rm -i -- \
  curl -s --max-time 10 http://demo-app.demo-app.svc.clusterset.local:9090
{"name":"demo-app","uri":"/","type":"HTTP","body":"Hello from demo-app","code":200}
The .svc.clusterset.local domain is provided by Submariner’s Lighthouse DNS. It resolves to the remote cluster’s service, proving end-to-end cross-cloud connectivity.

If the demo-app is not yet deployed on Cluster B, you can verify basic connectivity using the Submariner diagnostic tool:

subctl diagnose all --context cluster-a
Verify: Submariner status summary 
subctl show all --context cluster-a
CLUSTER ID                    ENDPOINT IP     PUBLIC IP       CABLE DRIVER   TYPE
cluster-a                     10.1.x.x        <aws-ip>        libreswan      local
cluster-b                     10.0.x.x        <azure-ip>       libreswan      remote

GATEWAY                       CLUSTER         REMOTE IP       NAT   CABLE DRIVER   SUBNETS                    STATUS   RTT avg
cluster-b-gateway-0           cluster-b       <azure-ip>       yes   libreswan      10.128.0.0/14, ...        connected   5.2ms

The connected status and RTT measurement confirm the tunnel is active between AWS and Azure.

What This Solves

Problem How Submariner Solves It

No network path between clouds

IPsec tunnel connects AWS VPC and Azure VNet at the pod/service CIDR level

Public IP exposure for cross-cluster traffic

All inter-cluster traffic flows through the encrypted tunnel - no public service endpoints needed

Manual VPN/peering configuration

ACM deploys and manages Submariner as an add-on - no cloud-specific VPN setup required

Cloud firewall complexity

Submariner automatically opens the required security group rules (AWS) and NSG rules (Azure)

Multi-cluster service discovery

Lighthouse DNS provides .svc.clusterset.local names for services across clusters

Alternatives Considered

Approach Notes

Submariner via ACM (this solution)

Managed as an ACM add-on, automatic cloud preparation, encrypted tunnels, integrated service discovery. Requires ACM.

Site-to-site VPN (Azure VPN Gateway + AWS VPN Gateway)

Cloud-native but requires manual setup on both sides, ongoing management, and does not provide service-level discovery.

Public ingress IPs

Simplest approach - expose services publicly and reference the public IPs. Works for demos but not acceptable for production (security, cost, latency).

VPC/VNet peering via cloud APIs

Requires same-provider or specific peering support. Does not work natively across Azure and AWS.

Skupper (Service Interconnect)

Application-layer connectivity (L7). Good for specific service-to-service links but does not provide the full L3 connectivity that Service Mesh needs for multi-cluster routing.

Reset

To undo everything from this step and return to the state before Submariner was deployed, run the following commands. This removes the add-ons (which uninstalls Submariner from both managed clusters), the SubmarinerConfigs, the Broker, the broker namespace, and the cloud credential secrets.

oc delete managedclusteraddon submariner -n cluster-a --context hub --ignore-not-found
oc delete managedclusteraddon submariner -n cluster-b --context hub --ignore-not-found
oc wait --for=delete managedclusteraddon/submariner -n cluster-a --context hub --timeout=300s 2>/dev/null || true
oc wait --for=delete managedclusteraddon/submariner -n cluster-b --context hub --timeout=300s 2>/dev/null || true
oc delete submarinerconfig submariner -n cluster-a --context hub --ignore-not-found
oc delete submarinerconfig submariner -n cluster-b --context hub --ignore-not-found
oc delete broker submariner-broker -n production-clusters-broker --context hub --ignore-not-found
oc delete namespace production-clusters-broker --context hub --ignore-not-found
oc delete secret cluster-a-cloud-creds -n cluster-a --context hub --ignore-not-found
oc delete secret cluster-b-cloud-creds -n cluster-b --context hub --ignore-not-found
After deleting the ManagedClusterAddOns, ACM will uninstall the Submariner operator and all its components (gateway, route agent, Lighthouse) from the managed clusters. The submariner-operator namespace on each managed cluster will be removed automatically. This process may take 2-3 minutes.
