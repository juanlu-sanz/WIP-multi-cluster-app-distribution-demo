= Step 0 - Current State: Manual Deployment Without Cross-Cluster Mobility
include::_attributes.adoc[]

This step reproduces the current operational reality: applications are deployed directly to individual clusters using `oc apply` or CI/CD pipelines that target a single cluster endpoint. There is no cross-cluster awareness - if an application needs to move from Cluster A to Cluster B (for upgrades, maintenance, or capacity), the team must manually reconfigure pipelines, recreate environments, and redirect DNS or load balancers. This is time-consuming, error-prone, and creates downtime windows.

== Prerequisites

* Two OpenShift {ocp-version} clusters (referred to as `cluster-a` and `cluster-b` in this demo)
* `oc` CLI authenticated as cluster-admin on both clusters
* Contexts renamed as described in the xref:00-setup.adoc[Cluster Setup] section

NOTE: In a real-world scenario without contexts configured, teams would `oc login` to each cluster individually and re-authenticate every time they switch. This demo uses `--context` flags for convenience, but the manual, per-cluster nature of the workflow is the same.

== Steps to Reproduce

[#deploy-cluster-a]
=== 1. Deploy the sample application to Cluster A

*What:* Deploy a stateless web application (link:https://github.com/stefanprodan/podinfo[podinfo]) directly to a single cluster using `oc apply`. Podinfo displays pod name, namespace, node, and other runtime details in its web UI - making it easy to see which cluster is serving traffic.

*Why:* This represents the current workflow - each cluster is targeted individually, with no central control plane aware of where the application lives.

[.console-input]
[source,bash,subs="+macros,+attributes"]
----
oc apply -f 00-current-state/namespace.yaml --context cluster-a
oc apply -f 00-current-state/deployment.yaml --context cluster-a
oc apply -f 00-current-state/service.yaml --context cluster-a
oc apply -f 00-current-state/route.yaml --context cluster-a
----

.Verify: Application running on Cluster A
[%collapsible]
====
[.console-input]
[source,bash]
----
oc get pods -n demo-app --context cluster-a
----

[.console-output]
[source,bash]
----
NAME                        READY   STATUS    RESTARTS   AGE
demo-app-6d4f5b7c8-x9k2m   1/1     Running   0          30s
----

[.console-input]
[source,bash]
----
oc get route demo-app -n demo-app --context cluster-a -o jsonpath='{.status.ingress[0].host}'
----

[.console-output]
[source,bash]
----
demo-app-demo-app.apps.cluster-a.example.com
----

Open the route URL in a browser to see the podinfo web UI displaying the pod name, namespace, and cluster details.

[.console-input]
[source,bash]
----
curl -sk https://$(oc get route demo-app -n demo-app --context cluster-a -o jsonpath='{.status.ingress[0].host}')/api/info
----

[.console-output]
[source,json]
----
{
  "hostname": "demo-app-6d4f5b7c8-x9k2m",
  "version": "6.x.x",
  "message": "demo-app (manual deployment)",
  "goos": "linux",
  "goarch": "amd64",
  ...
}
----

[#migrate-cluster-b]
=== 2. Simulate the need to migrate to Cluster B

*What:* Manually recreate the same application on a second cluster, simulating what happens when Cluster A needs maintenance or decommissioning.

*Why:* This exposes the core problem - there is no mechanism to "move" an application. The team must repeat the entire deployment from scratch on the new cluster and then manually update every external dependency (DNS, load balancers, CI/CD pipelines).

Assume Cluster A needs maintenance. The application must move to Cluster B.

*What the team currently has to do:*

. Target `cluster-b`
. Manually recreate all resources
. Update DNS / load balancer to point to the new cluster
. Delete from the old cluster

[.console-input]
[source,bash,subs="+macros,+attributes"]
----
oc apply -f 00-current-state/namespace.yaml --context cluster-b
oc apply -f 00-current-state/deployment.yaml --context cluster-b
oc apply -f 00-current-state/service.yaml --context cluster-b
oc apply -f 00-current-state/route.yaml --context cluster-b
----

.Verify: Application running on Cluster B
[%collapsible]
====
[.console-input]
[source,bash]
----
oc get pods -n demo-app --context cluster-b
----

[.console-output]
[source,bash]
----
NAME                        READY   STATUS    RESTARTS   AGE
demo-app-6d4f5b7c8-a3b4n   1/1     Running   0          25s
----

[.console-input]
[source,bash]
----
oc get route demo-app -n demo-app --context cluster-b -o jsonpath='{.status.ingress[0].host}'
----

[.console-output]
[source,bash]
----
demo-app-demo-app.apps.cluster-b.example.com
----

NOTE: The URL has changed. Any external system, load balancer, or DNS entry that pointed to Cluster A must now be manually updated. This is where downtime occurs.
====

'''

[#cleanup-cluster-a]
=== 3. Clean up Cluster A (manual step)

*What:* Manually delete all application resources from the old cluster after verifying the new one is working.

*Why:* Without a control plane managing placement, cleanup is manual. If forgotten, orphaned resources waste compute and create confusion about where the application is actually running.

Go back to Cluster A and remove the old deployment:

[.console-input]
[source,bash,subs="+macros,+attributes"]
----
oc delete -f 00-current-state/route.yaml --context cluster-a
oc delete -f 00-current-state/service.yaml --context cluster-a
oc delete -f 00-current-state/deployment.yaml --context cluster-a
oc delete -f 00-current-state/namespace.yaml --context cluster-a
----

.Verify: Application removed from Cluster A
[%collapsible]
====
[.console-input]
[source,bash]
----
oc get namespace demo-app --context cluster-a
----

[.console-output]
[source,bash]
----
Error from server (NotFound): namespaces "demo-app" not found
----
====

'''

== Why This Is a Problem

[cols="2,3",options="header"]
|===
| Problem | Impact

| No single control plane for placement
| Each cluster is managed independently - no way to declare "this app should run on Cluster B instead"

| DNS / Route changes on every migration
| External consumers must update their endpoints or wait for DNS propagation

| No traffic shifting
| Migration is all-or-nothing - no gradual rollover, no canary, no blue-green

| Pipeline reconfiguration
| CI/CD pipelines are hardcoded to a single cluster endpoint and must be manually updated

| No disaster recovery posture
| If a cluster goes down, there is no automated failover for stateless workloads

| Manual process at scale
| With 10+ clusters, this process becomes a significant operational burden
|===

The next steps in this demo address these problems:

* xref:02-acm-workload-placement.adoc[Step 1] introduces ACM to manage placement declaratively
* xref:03-submariner-connectivity.adoc[Step 2] deploys Submariner for encrypted cross-cloud connectivity between Azure and AWS
* xref:04-service-mesh.adoc[Step 3] adds Service Mesh for transparent traffic routing and zero-downtime failover

'''

== Reset

To return to the beginning of this step (before anything was created), delete the `demo-app` namespace from both clusters:

[.console-input]
[source,bash]
----
oc delete namespace demo-app --context cluster-a --ignore-not-found
oc delete namespace demo-app --context cluster-b --ignore-not-found
----
