= Step 1 - Solution: ACM Workload Placement with GitOps
include::_attributes.adoc[]

This step introduces Red Hat Advanced Cluster Management (ACM) combined with OpenShift GitOps (Argo CD) to manage workload placement declaratively. Instead of manually deploying to individual clusters and reconfiguring pipelines, we define a `Placement` rule that tells ACM where the application should run and an `ApplicationSet` that tells Argo CD what to deploy. To migrate the application from one cluster to another, we simply update the Placement - ACM and Argo CD handle the rest. The entire desired state lives in Git, providing a single control plane for multi-cluster deployments and a foundation for disaster recovery.

== Prerequisites

* ACM Hub cluster (OpenShift {ocp-version}) with ACM {acm-version} installed
* At least two managed clusters joined to the Hub (`cluster-a` and `cluster-b`)
* OpenShift GitOps Operator installed on the Hub
* `oc` CLI authenticated as cluster-admin on the Hub cluster
* Contexts renamed as described in the xref:00-setup.adoc[Cluster Setup] section
* Managed clusters labeled appropriately (see step 1 below)

NOTE: All commands in this step run on the Hub cluster (`--context hub`) unless a `--context cluster-a` or `--context cluster-b` flag specifies otherwise.

== How It Works

The push model uses four resources working together:

. *ManagedClusterSet + Binding* - Groups clusters and exposes them to the `openshift-gitops` namespace
. *Placement* - Selects which clusters receive the workload based on labels
. *GitOpsCluster* - Registers ACM managed clusters as Argo CD cluster secrets
. *ApplicationSet* - Reads from the PlacementDecision and generates an Argo CD Application per selected cluster

When the Placement changes (e.g. different region label), ACM updates the PlacementDecision. The ApplicationSet detects this, removes the Application from the old cluster (pruning all resources), and creates a new Application on the newly selected cluster.

== Steps to Apply

[#label-clusters]
=== 1. Label your managed clusters

*What:* Assign region and environment labels to each managed cluster registered in ACM.

*Why:* ACM Placement rules select target clusters based on labels. By labeling clusters with `region` and `environment`, we can write placement rules like "deploy to the cluster in west-europe" - and later change to "deploy to north-europe" with a single YAML edit.

[.console-input]
[source,bash,subs="+macros,+attributes"]
----
oc label managedcluster cluster-a region=west-europe environment=production --overwrite --context hub
oc label managedcluster cluster-b region=north-europe environment=production --overwrite --context hub
----

.Verify: Cluster labels applied
[%collapsible]
====
[.console-input]
[source,bash]
----
oc get managedclusters --show-labels --context hub
----

[.console-output]
[source,bash]
----
NAME        HUB ACCEPTED   MANAGED CLUSTER URLS                  JOINED   AVAILABLE   AGE   LABELS
cluster-a   true           https://api.cluster-a.example.com     True     True        10d   environment=production,region=west-europe,...
cluster-b   true           https://api.cluster-b.example.com     True     True        10d   environment=production,region=north-europe,...
----
====

[#create-clusterset]
=== 2. Create the ManagedClusterSet and bind it to the GitOps namespace

*What:* Group the managed clusters into a `ManagedClusterSet` and bind it to the `openshift-gitops` namespace where Argo CD runs.

*Why:* A ManagedClusterSet defines the pool of clusters that Placement rules can select from. The binding allows resources in `openshift-gitops` to reference this set. Without the binding, the Placement has no clusters to choose from.

[.console-input]
[source,bash,subs="+macros,+attributes"]
----
oc apply -f 01-acm-workload-placement/managedclusterset.yaml --context hub
oc apply -f 01-acm-workload-placement/managedclusterset-binding.yaml --context hub
----

.Verify: ManagedClusterSet created and bound
[%collapsible]
====
[.console-input]
[source,bash]
----
oc get managedclusterset production-clusters --context hub
----

[.console-output]
[source,bash]
----

----

[.console-input]
[source,bash]
----
oc get managedclustersetbinding -n openshift-gitops --context hub
----

[.console-output]
[source,bash]
----

[#register-clusters]
=== 3. Register clusters with Argo CD via GitOpsCluster

*What:* Create a Placement that selects all production clusters, and a GitOpsCluster CR that registers those clusters as Argo CD cluster secrets.

*Why:* Argo CD needs cluster credentials (stored as Kubernetes secrets) to deploy applications remotely. The GitOpsCluster CR bridges ACM's cluster inventory with Argo CD - any cluster selected by the referenced Placement gets a corresponding Argo CD secret. This is what enables Argo CD to push manifests to managed clusters.

[.console-input]
[source,bash,subs="+macros,+attributes"]
----
oc apply -f 01-acm-workload-placement/placement-all-clusters.yaml --context hub
oc apply -f 01-acm-workload-placement/gitopscluster.yaml --context hub
----

.Verify: Argo CD cluster secrets created
[%collapsible]
====
Wait 30-60 seconds for ACM to sync the cluster secrets.

[.console-input]
[source,bash]
----
oc get secrets -n openshift-gitops -l argocd.argoproj.io/secret-type=cluster --context hub
----

[.console-output]
[source,bash]
----
NAME                                          TYPE     DATA   AGE
cluster-a-cluster-secret                      Opaque   3      30s
cluster-b-cluster-secret                      Opaque   3      30s
----

Also verify the `acm-placement` ConfigMap exists (required by the ApplicationSet generator):

[.console-input]

[#deploy-appset]
=== 4. Create the Placement and ApplicationSet to deploy the application

*What:* Create a Placement that targets `cluster-a` (west-europe) and an ApplicationSet that deploys the application to whichever clusters that Placement selects.

*Why:* This is the core of the GitOps-based deployment model. The Placement tells ACM which cluster should run the application. The ApplicationSet uses a `clusterDecisionResource` generator that reads from ACM's PlacementDecision - it knows which clusters are selected and generates one Argo CD Application per cluster. Argo CD then syncs the application manifests from the `kustomize/overlays/production` path in Git to the target cluster.

[.console-input]
[source,bash,subs="+macros,+attributes"]
----
oc apply -f 01-acm-workload-placement/placement.yaml --context hub
envsubst < 01-acm-workload-placement/applicationset.yaml | oc apply --context hub -f -
----

.Verify: Application deployed to Cluster A via Argo CD
[%collapsible]
====
Check the ApplicationSet and generated Application:

[.console-input]
[source,bash]
----
oc get applicationsets -n openshift-gitops --context hub
----

[.console-output]

[.console-input]
[source,bash]
----

[#migrate]
=== 5. Migrate the application to Cluster B

*What:* Patch the Placement to change the label selector from `region: west-europe` to `region: north-europe`, moving the application to cluster-b.

*Why:* This is the migration itself - and the key difference from Step 0. Instead of logging into two clusters, re-applying manifests, updating DNS, and cleaning up the old deployment, we patch a single Placement resource. ACM re-evaluates, updates the PlacementDecision, and the ApplicationSet automatically removes the Application from cluster-a (pruning all resources) and creates a new Application on cluster-b.

[.console-input]
[source,bash,subs="+macros,+attributes"]
----
oc patch placement demo-app-placement -n openshift-gitops --context hub --type merge \
  -p '{"spec":{"predicates":[{"requiredClusterSelector":{"labelSelector":{"matchLabels":{"region":"north-europe"}}}}]}}'
----

The Placement re-evaluates. Since the selector now matches `region: north-europe`, ACM updates the PlacementDecision:

* Clusters with `region: west-europe` are *removed* from the decision
* Clusters with `region: north-europe` are *added* to the decision

The ApplicationSet detects the PlacementDecision change and:

. Deletes the `demo-app-cluster-a` Application (Argo CD prunes all resources from cluster-a)
. Creates a new `demo-app-cluster-b` Application (Argo CD syncs manifests to cluster-b)

.Verify: Application migrated to Cluster B
[%collapsible]
====
Check the updated PlacementDecision:

[.console-input]
[source,bash]

  -l cluster.open-cluster-management.io/placement=demo-app-placement \
  -o jsonpath='{range .items[*]}{.status.decisions[*].clusterName}{"\n"}{end}'
----

----

[#automatic-failover]
=== 6. (Bonus) Automatic failover on cluster fault

*What:* When a managed cluster becomes unreachable, ACM automatically evicts it from the PlacementDecision, and the ApplicationSet redeploys the application on another healthy cluster - no manual intervention required.

*Why:* This provides disaster recovery for stateless workloads out of the box. Unlike the deprecated Channel/Subscription model, the ApplicationSet push model reacts to ACM taints in real-time. If cluster-a goes down, ACM marks it with a `cluster.open-cluster-management.io/unreachable` taint, the Placement evicts it (since our Placement does not tolerate that taint), and the ApplicationSet moves the application to the next matching cluster.

NOTE: This is automatic for stateless workloads only. Stateful applications with persistent data require additional tooling (OpenShift DR, VolSync, or OADP) and user-initiated failover to prevent data loss.

TIP: To delay automatic eviction (avoiding reaction to brief network disruptions), add a `tolerationSeconds` field to the Placement's taint tolerations. See the link:https://docs.redhat.com/en/documentation/red_hat_advanced_cluster_management_for_kubernetes/2.11/html/clusters/cluster_mce_overview#placement-tolerations[Placement tolerations documentation].

== What This Solves (Compared to Step 0)

[cols="2,3",options="header"]
|===
| Problem from Step 0 | How ACM + GitOps Solves It

| No single control plane
| ACM Hub + Argo CD manages all clusters from one place

| Manual pipeline reconfiguration
| Placement rules are declarative - patch the label selector, ApplicationSet handles the rest

| No disaster recovery posture
| Automatic failover when a cluster becomes unreachable; all state lives in Git

| Manual process at scale
| One `Placement` change moves the app across any number of clusters

| No GitOps workflow
| Argo CD syncs from Git with self-healing and automated pruning
|===

== Alternatives Considered

[cols="2,4",options="header"]
|===
| Approach | Notes

| ACM + GitOps ApplicationSet (this solution)
| Declarative, GitOps-native, single control plane for placement across all clusters. Automatic failover for stateless workloads. Requires ACM Hub + OpenShift GitOps.

| Manual `oc apply` per cluster
| Current state (Step 0). Simple but does not scale and creates operational burden.

| ACM Channel/Subscription (deprecated)
| Legacy ACM application lifecycle model. Deprecated since ACM 2.10 in favor of GitOps integration. Does not provide Argo CD self-heal or prune capabilities.

| Hive + ClusterDeployments
| Useful for cluster provisioning, not specifically for application placement.
|===
